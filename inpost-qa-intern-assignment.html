QA Technical Assignment

Welcome to the InPost QA internship assignment. You'll explore a real web application, report what you find, and optionally write automated tests against it.

There's no single correct answer here. We're looking at how you think, how you communicate, and how you approach problems — not just whether you know the right syntax.

AI tools are fully allowed — ChatGPT, Claude, Copilot, Cursor, anything you use day to day. You're responsible for every line you submit, so be ready to explain your reasoning.
How this assignment is scored
Part 1
Manual testing — find bugs in the app and write them up in test-report.md. The more bugs you find and the clearer your descriptions, the higher your score.
Part 2
Automation tasks — six tasks, all optional. Each task is scored 0 / 1 / 3 / 5 based on quality. Do as many as you can — a solid attempt at fewer tasks beats a rushed attempt at all of them.

We review every submission. Candidates with the highest scores will be contacted for the next stage. You'll hear back from us regardless of the outcome.

Prerequisites

Make sure you have the following installed before you start.

  • Node.js 18+ — nodejs.org  ·  download the LTS version
  • Git — git-scm.com
  • Code editor — VS Code recommended: code.visualstudio.com
  • GitHub account — github.com  ·  you'll need one to submit your work

Step-by-step guide

Step 1

Set up the web application

Clone the InPost web app and run it locally.

git clone [REPO_URL_HERE]
cd inpost-qa-site
npm install
npm run dev

Open your browser and go to http://localhost:3000 — you should see the InPost homepage.

Login credentials:

  • Email: user@example.com
  • Password: password12345
Keep this terminal window open. The app needs to be running for the rest of the assignment.
Step 2

Set up your test project

Download the test project as a ZIP file — this will become your submission repository.

  1. Go to [PLAYWRIGHT_REPO_URL_HERE]
  2. Click Code → Download ZIP
  3. Extract the ZIP and rename the folder to inpost-qa-firstname-lastname (e.g. inpost-qa-jan-kowalski)

Then install the dependencies and Playwright browsers:

cd inpost-qa-jan-kowalski
npm install
npx playwright install

Verify everything works:

npx playwright test

You should see 0 passed or the runner starting without errors. If you see a browser open briefly — that's normal.

The tasks/ folder contains 6 task files, each with instructions inside. You don't need to do all of them — see Step 5.
Step 3

Explore the app and write your test report

Explore http://localhost:3000 thoroughly — click around, try logging in, test edge cases. Focus on these three pages:

  • / — Home page
  • /login — Login page
  • /profile — Profile page (try visiting it both logged in and logged out)

Write down everything you find that looks wrong, broken, or unexpected. This is your test report.

Test report format

Create a file called test-report.md in the root of your repo. Organise your findings by page. The exact format is up to you — but it should be clear enough that a developer could understand what to fix.

You can include screenshots — put them in a screenshots/ folder and reference them in the report like this: ![description](screenshots/filename.png). A screenshot often says more than a paragraph of text.

test-report.md — example structure 
# Test Report — Online Shop QA

## Product page (/product)

**[High]** "Add to cart" does nothing when no size is selected
Clicking the Add to Cart button without choosing a size silently fails —
no error message, no feedback, cart count stays the same.
![No feedback on add to cart](screenshots/add-to-cart-silent.png)

**[Medium]** Price shows as "£NaN" on initial page load
Before any size option is selected, the price field shows "£NaN"
instead of a default price or a prompt to select a size.

**[Low]** Product thumbnails have no alt text
All product images in the gallery are missing alt attributes.
They appear as blank to screen readers and if images fail to load.

---

## Checkout page (/checkout)

**[High]** Applying an invalid discount code shows no error
Entering a made-up code and clicking Apply does nothing visible —
no error, no confirmation. It's unclear whether the code was rejected.
![Discount code no feedback](screenshots/discount-no-feedback.png)

---

## Order history page (/orders)

**[Medium]** Order dates show as "undefined/undefined/undefined"
All past orders display the date as "undefined/undefined/undefined"
instead of the actual order date.

Use severity labels that make sense to you — High for things that break core functionality, Medium for noticeable issues, Low for minor cosmetic problems.

The more specific you are, the better. "Something is broken" is less useful than "clicking the search button returns a 500 error message". You don't need to explain why it's broken — just describe what you see.
Step 4

Automation tasks — optional, all tasks earn points

The tasks/ folder has 6 task files. All are optional — do as many as you can. Each task you complete earns points based on the quality of your solution.

0 pts — not attempted or doesn't run 1 pt — basic, works 3 pts — solid, well structured 5 pts — excellent
  • 01
    Login

    Write tests for the login flow. Test the happy path and error cases. Verify the user is actually logged in after a successful login — not just that the URL changed.

    max 5 pts
  • 02
    God Test Refactor

    You're given a working but badly written test. Identify what's wrong with it and refactor it. The instructions are in the task file.

    max 5 pts
  • 03
    Bad Selectors / Debug

    Six tests — some failing, some not. Find what's wrong with each one and fix it. One of them is the hardest to spot.

    max 5 pts
  • 04
    Async Wait

    Write a stable test for the parcel tracking page at /challenges/async. The test must pass consistently across multiple runs — no hardcoded delays.

    max 5 pts
  • 05
    Visual Testing

    Write a visual regression test for the locker details page at /challenges/visual. The data on this page changes on every load — you'll need to handle that.

    max 5 pts
  • 06
    API Testing

    Write a full API test suite for the parcel endpoints. The documentation and a sandbox token are available at /challenges/api-testing.

    max 5 pts
If you're new to Playwright, start with Task 01 — it's the most approachable. The Playwright docs are well written and the AI tools you already use can help you get started quickly.
Step 5

Push to GitHub and submit your work

Create a new public GitHub repository and push your project there.

  1. Go to github.com → click New repository
  2. Name it inpost-qa-firstname-lastname (e.g. inpost-qa-jan-kowalski)
  3. Set visibility to Public → click Create repository

Make sure your repository contains:

  • test-report.md — your manual test findings, organised by page
  • tasks/ — task files with your solutions (even partial attempts are fine)
  • playwright.config.ts — pointing to http://localhost:3000
  • package.json

Initialise git and push everything:

git add .
git commit -m "QA assignment submission"
git push
Commit and push regularly as you progress — not just at the end.
