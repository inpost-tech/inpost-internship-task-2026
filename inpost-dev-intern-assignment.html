Developer Technical Assignment

We'd love to get to know you better — not just your coding skills, but also how you think, communicate, and approach problems that don't come with a step-by-step manual.

There's no single correct solution. We want to see your interpretation, your decisions, and your ability to deliver something that works and makes sense.

AI tools are fully allowed — ChatGPT, Claude, Copilot, Cursor, anything you use day to day. But if an AI wrote your code, you should understand every line of it and be able to explain your decisions. We may ask.

The Task

InPost operates a network of over 90,000 parcel pick-up points across Europe. Data about every single one of them is available through a REST API:

GET https://api-global-points.easypack24.net/v1/points

Returns a paginated list of all InPost points across Europe — including location coordinates, address details, available functions, operating hours, locker availability, and more.

Explore the API yourself. The shape of the data should inform what you build.

Using this API as your primary data source, build something useful.

That's it. The rest is up to you.

What we mean by "something useful"

This could be a backend service, a CLI tool, a data pipeline, an analytics module, a web application, or anything else you find interesting. Here are a few directions to inspire you — pick one, combine several, or ignore them entirely:

🔍 Smart finder

Help users find the best parcel locker for their needs based on various criteria

📊 Distribution analysis

Analyze the density and coverage of parcel lockers across regions

📡 Status monitor

Track and report on locker availability and operational status

🔧 API enrichment

Wrap, reshape, or aggregate the data into something more useful

🗺️ Personal problem

Solve a problem you personally experienced when using parcel lockers

💡 Your own idea

Something we haven't thought of — surprise us

We value quality over quantity. A well-designed, well-tested solution to a small, clearly defined problem will always beat a shallow attempt to do everything. Show us that you can make deliberate choices about what to build — and what to leave out.

Requirements

  1. Use any programming language and tech stack you're comfortable with. Pick whatever lets you show your best work.
  2. Your solution must actually interact with the InPost API — fetch real data, process it, do something meaningful with it.
  3. Publish your code in a public GitHub repository. The repo should contain everything needed to build and run your solution.
  4. Fill in the README.md template and include it the root directory of Github repository with your solution. It's your main way of communicating with us — treat it as a conversation, not a formality. Fill all provided sections but if you want to share more information, don't hesitate to add more.
  5. Send us the link to your repository until 6 May (end of the day).

What we're looking for

We'll be reading your code and your README.md. Here's what matters to us:

What Why
Clear thinking & communication Can you explain what you built and why? Is your README.md a pleasure to read?
Problem definition The task is intentionally vague. How you narrow it down into a concrete, solvable problem matters more than what you pick.
Working software It should build. It should run. It should do what your README.md says it does.
Code quality Clean, readable, idiomatic code. Reasonable project structure. Meaningful names.
Technical decisions Did you make deliberate choices about architecture, libraries, data handling, error cases? Can you justify them?
Testing Tests are not required, but if you provide any way of automated testing, it is a nice bonus.
Creativity & initiative Did you go beyond the minimum? Surprise us. Show us something we didn't expect.

A few tips

  • Start with the API. Explore it. Understand the data. The shape of the data should inform what you build.
  • Scope wisely. You don't need to use every field from the API. Pick a focused slice and do it well.
  • Don't over-engineer. A clean, simple solution that works is better than an over-abstracted one that doesn't.
  • Have fun with it. This task is designed to give you space to be creative. Use it.

FAQ

Can I use external libraries / frameworks?

Yes, absolutely. Just make sure your build instructions work.

Do I need to deploy it somewhere?

It is not necessary, but if you do deploy it, that's a nice bonus. If you don't deploy, make sure to include screenshots, a short video, or example outputs in your README.md so we can see how your solution works.

Can I ask questions?

The intentional ambiguity is part of the exercise. Make reasonable assumptions, document them in your README.md, and move forward. Figuring out what needs clarification is part of the task.
